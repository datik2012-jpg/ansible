This playbook automates adding a firewall rule using the Firewalld module in Ansible.

---
- name: Add Firewalld rule
  hosts: all
  tasks:
    - firewalld:
        port: 8080/tcp
        service: http
        source: 192.0.0.0/24
        zone: public
        state: enabled
        permanent: yes
        immediate: yes




ey	Meaning
name	Describes the playbook’s purpose — here, adding a Firewalld rule.
hosts: all	Applies the task to all hosts defined in your inventory.
tasks	Lists the actions Ansible will perform.
firewalld:	Uses the built-in Ansible module to manage Firewalld rules.
port: 8080/tcp	Opens TCP port 8080.
service: http	Associates the rule with the HTTP service.
source: 192.0.0.0/24	Restricts access to the specified IP range.
zone: public	Applies the rule to the “public” zone in Firewalld.
state: enabled	Ensures the rule is active.
permanent: yes	Makes the rule persist after reboot.
immediate: yes	Applies the rule instantly without needing a reload.


here saved permament: yes:

When you set permanent: yes, Ansible tells Firewalld to store the rule in its persistent configuration files, not just in memory.

Location:  
/etc/firewalld/zones/  
Each zone (like public.xml) contains XML definitions of ports, services, and sources.

Mechanism:  
Firewalld maintains two configurations:

When permanent: yes and immediate: yes are both set, Ansible applies the rule immediately to the runtime configuration and writes it to the permanent XML file so it survives reboot.


Platform	Persistence Mechanism	File/Registry Path
Linux (Firewalld)	XML zone files	/etc/firewalld/zones/
Windows	Registry entries	HKLM\SYSTEM\CurrentControlSet\Services\SharedAccess\Parameters\FirewallPolicy\

This YAML snippet is an Ansible playbook that performs two complementary tasks — compressing and uncompressing a folder. It’s a great example of how Ansible automates file packaging and restoration across multiple hosts.

- hosts: all
  tasks:
    - name: Compress a folder
      archive:
        path: /opt/app/web
        dest: /tmp/web.gz
        format: zip|tar|bz2|xz|gz

    - name: Uncompress a folder
      unarchive:
        src: /tmp/web.gz
        dest: /opt/app/web
        remote_src: yes



In your playbook, both the archive and unarchive modules run on each target host, not on your local control machine (unless you explicitly target localhost).

So:

When you run the playbook with hosts: all, Ansible connects to each remote server via SSH.

The compression (archive) happens on that remote server — the folder /opt/app/web is compressed locally on that host.

The resulting file /tmp/web.gz is created on that same remote host.

Another example:

compress on your local machine (localhost) and then copy the archive to multiple remote hosts using Ansible.


Step 1 — Compress on localhost
Run a play that targets only your control node:

yaml
- hosts: localhost
  tasks:
    - name: Compress folder locally
      archive:
        path: /opt/app/web
        dest: /tmp/web.tar.gz
        format: gz
This creates /tmp/web.tar.gz on your local machine.

✅ Step 2 — Copy the archive to multiple hosts
Now send the archive to all remote servers:

yaml
- hosts: all
  tasks:
    - name: Copy archive to remote hosts
      copy:
        src: /tmp/web.tar.gz
        dest: /tmp/web.tar.gz
This transfers the file from localhost → each remote host.

✅ Step 3 — Uncompress on each remote host (optional)
If you want to extract it on each server:

yaml
- hosts: all
  tasks:
    - name: Uncompress archive on remote hosts
      unarchive:
        src: /tmp/web.tar.gz
        dest: /opt/app/web
        remote_src: yes
remote_src: yes tells Ansible the file already exists on the remote host, so it won’t try to copy it again.

  tasks:
    - name: Configure ssh keys
      authorized_keys:
        user: maria
        state: present
        key: |
          ssh-rsa AAAAB3NzaC1yc2EAAAAADAQABAAA...
          BAQC4WKn4K2G3iWg9HdCGo34gh+……root@97a1b9c3a


This section contains the public SSH key that you want to add to the user’s ~/.ssh/authorized_keys file.

The | symbol means “multiline string” in YAML — it lets you paste the entire key exactly as it appears.

The key itself (starting with ssh-rsa) is already generated elsewhere — typically using ssh-keygen on your local machine or another trusted system.

Ansible doesn’t generate the key here; it simply copies it to the remote host.

Key	Meaning
hosts: all	Runs this task on all hosts in your inventory.

user: maria	Specifies the user whose ~/.ssh/authorized_keys file will be modified.
state: present	Ensures the key is added (creates the file if missing).
key:	The actual SSH public key to insert. The `	` allows multiline input.


Security Model:


Option 1 — One public key for all servers
This is the simplest and most common setup in automation.

How it works
You generate one key pair (private + public) on your workstation:

bash
ssh-keygen -t rsa -b 4096
Then you distribute the same public key to all servers:

yaml
authorized_keys:
  user: maria
  key: |
    ssh-rsa AAAA...
When this is good
You want one identity to access all servers

You want easy automation (CI/CD, Ansible control node, Jenkins, GitLab runners)

You want simple key rotation

You trust the environment and access is controlled

Security trade‑off
If the private key is compromised, the attacker can access all servers.



Option 2 — A different public key per server
This is more secure but more work.

How it works
You generate multiple key pairs, one per server or per user:

Code
id_rsa_server1.pub
id_rsa_server2.pub
id_rsa_server3.pub
Then you assign each key to the correct host using inventory variables:

yaml
authorized_keys:
  user: maria
  key: "{{ hostvars[inventory_hostname].ssh_key }}"
When this is good
You want per‑server isolation

You want per‑user isolation

You want zero trust between servers

You want fine‑grained revocation (remove one key without affecting others)

Security trade‑off
More keys = more complexity.

🧠 Which one should YOU use?
Most organizations use:

One key per user → installed on all servers

Separate keys for automation (Ansible, CI/CD)

Separate keys for privileged accounts (root, admin)

This gives a balance of security and simplicity.

On node00 we want to keep the installed packages up to date, so we would like to run yum updates regularly. Create a playbook ~/playbooks/yum_update.yml and create a cron job as described below:

a. Do not add cron directly using crontab instead create a cron file /etc/cron.d/ansible_yum.

d. Cron should be added for user root


Use command yum -y update

from examples:ansible-doc ansible.builtin.cron:


[thor@ansible-controller playbooks]$ cat yum_update.yml

  hosts: node00
  become: yes
  tasks:
    - name: Cron file creation at /etc/cron.d/ansible_yum
      ansible.builtin.cron:
        name: "yum update"
        weekday: "0"
        minute: "5"
        hour: "8"
        user: root
        job: "yum -y update"
        cron_file: ansible_yum
