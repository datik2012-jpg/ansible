Roles
Roles are structured, reusable collections of tasks, variables, templates, and handlers.
They’re designed for organization and reuse across multiple playbooks.

Example:
yaml
- hosts: webservers
  roles:
    - apache
    - firewall
Folder structure:
Code
roles/
  apache/
    tasks/
      main.yml
    templates/
      httpd.conf.j2
    vars/
      main.yml
✅ Why use roles
Encapsulate logic for reuse

Automatically load tasks, vars, handlers, templates

Cleaner playbooks

Easier maintenance and sharing

Confirms your inventory syntax is valid.

Shows how Ansible merges multiple inventory sources.

Helps debug host grouping and variable inheritance.




Includes are inline task imports — they pull in another YAML file dynamically.

Example:
yaml
- name: Include web tasks
  include_tasks: web.yml
or older syntax:

yaml
- include: web.yml
✅ Why use includes
Useful for splitting large playbooks into smaller files

Allow conditional inclusion (when: statements)

More flexible than roles for ad‑hoc task grouping

🔍 Key differences
Feature	Roles	Includes
Purpose	Reusable, structured automation	Inline task inclusion
Organization	Has folders (tasks, vars, handlers, templates)	Single YAML file
Reuse	High — can be shared across playbooks	Limited — local to playbook
Variables	Automatically loaded	Must be defined manually
Conditional use	Harder (requires tags or vars)	Easy (when: supported)
Best for	Modular design	Breaking up large playbooks

Note: Option is create from your yml files ROLES and put in galaxy folder structure (even if you do not want to upload it to the site galaxy):


Core idea
Ansible Galaxy does not run random .yml files.
Galaxy installs roles (or collections), and roles have a fixed structure.

So your two YAMLs must become:

role tasks

inside a role directory

with a tasks/main.yml that includes your two YAMLs

🧩 Step 1 — Create a role
Use Galaxy to create the role skeleton:

bash
ansible-galaxy init fullstack
This creates:

Code
fullstack/
  tasks/
    main.yml
  handlers/
  vars/
  defaults/
  templates/
  files/
🧩 Step 2 — Move your two YAMLs into the role
Assume your two playbooks are:

install-db.yml

install-web.yml

Move them into the role’s tasks/ folder:

Code
fullstack/tasks/install-db.yml
fullstack/tasks/install-web.yml
🧩 Step 3 — Include them from main.yml
Edit:

Code
fullstack/tasks/main.yml
Put:

yaml
---
- name: Install DB server
  import_tasks: install-db.yml

- name: Install Web server
  import_tasks: install-web.yml
Now your role is fully modular.

🧩 Step 4 — Use the role in your main playbook
Your main playbook becomes:

yaml
---
- name: Configure full stack server
  hosts: f-server-1
  roles:
    - fullstack
That’s it — Galaxy can now install and use this role.

⭐ Why this works
Galaxy installs roles, not raw YAML files.
By converting your two YAMLs into a role:

You get a reusable module

You can publish it

You can install it with ansible-galaxy install

You can version it

You can share it with your team

You get clean structure and best practices

📦 Optional: Publish your role to Galaxy
If you want others to use it:

bash
ansible-galaxy role import your_github_username fullstack
🔥 Summary
To use your two YAMLs with Galaxy:

Create a role

Put your two YAMLs inside tasks/

Include them from tasks/main.yml

Call the role from your playbook

This is the correct and clean way to integrate your existing playbooks with Galaxy.



ole management
Command	Description
ansible-galaxy list	Shows all installed roles.
ansible-galaxy install <role_name>	Installs a role from Galaxy or GitHub.
ansible-galaxy install <role_name> -p ./roles	Installs a role into a custom path.
ansible-galaxy remove <role_name>	Removes an installed role.
ansible-galaxy init <role_name>	Creates a new role skeleton.
ansible-galaxy role info <role_name>	Displays metadata and details about a role.
ansible-galaxy role search <keyword>	Searches Galaxy for roles matching a keyword.
ansible-galaxy role import <namespace> <role_name>	Imports a role from GitHub into Galaxy.
ansible-galaxy role setup	Prepares a role for publishing (used internally).


📦 Collection management
Command	Description
ansible-galaxy collection list	Lists installed collections.
ansible-galaxy collection install <namespace.collection>	Installs a collection.
ansible-galaxy collection remove <namespace.collection>	Removes a collection.
ansible-galaxy collection init <namespace.collection>	Creates a new collection skeleton.
ansible-galaxy collection build	Packages a collection into a .tar.gz file.
ansible-galaxy collection publish <path_to_tar>	Publishes a collection to Galaxy.
ansible-galaxy collection verify <path_to_tar>	Verifies a collection archive before publishing.
