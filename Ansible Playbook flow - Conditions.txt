Conditions in loops

---
- name: Install Softwares
  hosts: all
  vars:
    packages:
      - name: nginx
        required: True
      - name: mysql
        required: True
      - name: apache
        required: False

  tasks:
    - name: Install "{{ item.name }}" on Debian
      apt:
        name: "{{ item.name }}"
        state: present
      when: item.required == True
      loop: "{{ packages }}"

Explanation
Playbook name: “Install Softwares” — defines the purpose of the playbook.

Hosts: all — runs on all target machines.

Variables: packages — a list of software items with a required flag.

Task: Installs each package using the apt module (Debian-based systems).

Condition: Only installs packages where required == True.

Loop: Iterates through the packages list.

This playbook will install nginx and mysql, skipping apache because its required flag is False.

In case we need stop the execution of the playbook when at least one step failed for any sevrer: any_errors_fatal: true

- name: Deploy web application
  hosts: server1,server2,server3
  any_errors_fatal: true
  tasks:


Concept	Explanation
hosts	Lists all target machines (server1, server2, server3). Ansible automatically runs tasks on each host.
any_errors_fatal	If any host fails, Ansible stops the entire play immediately — no other hosts continue.
tasks	Placeholder for the actions you’ll define next (e.g., deploy files, restart services).


Why use any_errors_fatal: true
Normally, Ansible continues running on other hosts even if one fails.
Setting this flag ensures atomic deployment — either all servers succeed, or the play stops.
That’s ideal for web app rollouts where partial success could break consistency.


Another example 30% of servers if fails stop the execution:

- name: Deploy web application
  hosts: server1,server2,server3
  max_fail_percentage: 30


hosts	Targets the listed servers (server1, server2, server3).
max_fail_percentage	Allows up to 30 % of hosts to fail before Ansible stops the play. If more than 30 % fail, execution halts for all remaining hosts.

            to: admin@company.com
            subject: Playbook Failed



Concept	Description
block	Groups tasks together so they share error handling logic.
rescue	Executes only if any task inside the block fails.
any_errors_fatal	Stops the play entirely if any host fails — ensures consistency across all servers.
mail	Sends an email notification when the block fails.


So, if any of the installation or configuration tasks fail, the rescue block triggers and sends an email to admin@company.com with the subject “Playbook Failed.”

/////////////////////////////////Ansible Parallelism - LAB ///////////

Which of the following is not a valid strategy in Ansible?


Valid Ansible strategies
These are real, supported strategies in Ansible:

linear — default, runs hosts one‑by‑one

free — hosts run tasks independently (non‑blocking)

host_pinned — keeps tasks pinned to the same worker

debug — pauses after each task for inspection

serial — not a strategy, but often confused with one

serial is a play-level keyword, not a strategy.

It controls how many hosts run at the same time, for example:

yaml
serial: 2


❌ Invalid strategies
These do NOT exist in Ansible:

rolling — people confuse this with serial

parallel — not a strategy

batch — not a strategy


The real Ansible strategies
These are the only valid strategy plugins:

linear — default, runs hosts in order

free — hosts run tasks independently

debug — pauses after each task

host_pinned — keeps a host on the same worker

[defaults]
forks = 5
Meaning:

Ansible can run up to 5 hosts in parallel maximum, regardless of serial.

A:Linear
