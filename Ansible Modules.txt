Ansible lineinfile approach
yaml
- name: Add DNS server to resolv.conf
  hosts: localhost
  tasks:
    - lineinfile:
        path: /etc/resolv.conf
        line: 'nameserver 10.1.250.10'
Behavior: Searches for the line first; adds it only if missing.

Result: File stays clean — no duplicates.

Advantage: Idempotent and declarative (Ansible ensures the desired state).

Extra power: You can use regex to replace or match patterns.



Basic example
yaml
- name: Ensure nameserver entry exists
  hosts: localhost
  become: yes
  tasks:
    - lineinfile:
        path: /etc/resolv.conf
        line: 'nameserver 10.1.250.10'
        state: present
🔍 What it does
Searches /etc/resolv.conf for the line nameserver 10.1.250.10.

If it’s missing, it adds it.

If it’s already there, it does nothing.

This makes the task idempotent — safe to run multiple times.

🧩 Search and replace using regex
If you want to replace a line that matches a pattern:

yaml
- name: Replace existing nameserver line
  lineinfile:
    path: /etc/resolv.conf
    regexp: '^nameserver'
    line: 'nameserver 10.1.250.10'
✔ Finds any line starting with nameserver  
✔ Replaces it with your new line

🧩 Add after or before a specific line
yaml
- name: Add nameserver after a specific line
  lineinfile:
    path: /etc/resolv.conf
    line: 'nameserver 10.1.250.10'
    insertafter: '^nameserver 10.1.250.2'
Or before:

yaml
insertafter: '^nameserver 10.1.250.1'
insertbefore: '^search'
🧩 Add multiple lines
yaml
- name: Add multiple nameservers
  lineinfile:
    path: /etc/resolv.conf
    line: "{{ item }}"
    state: present
  loop:
    - 'nameserver 10.1.250.10'
    - 'nameserver 10.1.250.11'

///////////////////////ANSIBLE PLUGINS /////////


Dynamic Inventory Plugin

Used to generate inventory dynamically instead of using a static file.


xample:
AWS EC2 dynamic inventory plugin (aws_ec2):

yaml
plugin: aws_ec2
regions:
  - us-east-1
keyed_groups:
  - key: tags.Name
    prefix: tag
✅ Automatically pulls hosts from AWS based on tags or regions.
✅ Other examples: azure_rm, gcp_compute, openstack, vmware_vm_inventory.


Module Plugin
Modules are the units of work — they perform actions on remote hosts.

🔹 Example:
yum module:

yaml
- name: Install Apache
  yum:
    name: httpd
    state: present
✅ Executes commands remotely.
✅ Other examples: copy, file, service, user, shell, command.


Action Plugin
Action plugins extend or modify how modules run — they execute on the control node before sending tasks to remote hosts.

🔹 Example:
normal action plugin (default behavior):

yaml
- name: Run a custom action
  action: my_custom_action
Or a custom plugin that checks parameters before running a module:

python
# custom_action.py
from ansible.plugins.action import ActionBase

class ActionModule(ActionBase):
    def run(self, tmp=None, task_vars=None):
        self._display.display("Running custom validation before module execution")
        result = super(ActionModule, self).run(tmp, task_vars)
        return result
✅ Used for pre‑processing, validation, or combining multiple modules.
✅ Examples: template, copy, and assemble use internal action plugins.




ookup Plugins
Used to fetch data from external sources (files, environment variables, URLs, etc.) during playbook execution.

Example:
yaml
- name: Read a local file
  debug:
    msg: "{{ lookup('file', '/etc/hosts') }}"
✅ Reads the file on the control node.
Other examples: env, pipe, url, ini, password.

🎛️ Filter Plugins
Used to transform or manipulate data inside Jinja2 expressions — like formatting, sorting, or converting.

Example:
yaml
- name: Show uppercase fruit names
  vars:
    fruits: ['apple', 'banana', 'grapes']
  debug:
    msg: "{{ fruits | map('upper') | list }}"
✅ Converts all fruit names to uppercase.
Other examples: default, join, unique, dict2items.

🔌 Connection Plugins
Define how Ansible connects to remote systems — SSH, local, Docker, etc.

Example:
yaml
- name: Run locally
  hosts: localhost
  connection: local
✅ Runs tasks on the control node instead of over SSH.
Other examples: ssh, paramiko, docker, winrm.

🗂️ Inventory Plugins
Used to build inventory dynamically from external sources (clouds, databases, APIs).

Example:
yaml
plugin: aws_ec2
regions:
  - us-east-1
keyed_groups:
  - key: tags.Name
    prefix: tag
✅ Automatically discovers EC2 instances and groups them.
Other examples: azure_rm, gcp_compute, openstack, yaml.

📞 Callback Plugins
Control how Ansible displays output or sends notifications (to Slack, JSON, etc.).

Example:
Enable the json callback:

bash
ANSIBLE_STDOUT_CALLBACK=json
✅ Makes Ansible output structured JSON instead of plain text.
Other examples: minimal, yaml, profile_tasks, slack, mail.

🧠 Summary Table
Plugin Type	Purpose	Example
Lookup	Fetch external data	lookup('file', '/etc/hosts')
Filter	Transform data	`{{ list	join(',') }}`
Connection	Define how to connect	connection: local
Inventory	Build dynamic inventory	plugin: aws_ec2
Callback	Customize output or notifications	ANSIBLE_STDOUT_CALLBACK=yaml

Which of the following Ansible modules support free_form parameter?

Example: Example:

yaml
command: ls -l /etc
Here, ls -l /etc is a free‑form parameter.

There is no cmd: or args: — the module expects a single free‑form string.

Modules that use free‑form parameters
1️⃣ command module
yaml
- command: echo "Hello dani"
2️⃣ shell module
yaml
- shell: cat /etc/resolv.conf | grep nameserver
3️⃣ raw module
yaml
- raw: uptime
4️⃣ git module (free‑form URL)
yaml
- git: https://github.com/example/repo.git
5️⃣ service module (older syntax)
yaml
- service: httpd state=started
6️⃣ mount module (older syntax)
yaml
- mount: /mnt src=/dev/sdb1 fstype=ext4 state=mounted
